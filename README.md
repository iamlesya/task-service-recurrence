# Task Service

Сервис для управления задачами с HTTP API на Go.

## Требования

- Go `1.23+`
- Docker и Docker Compose

## Быстрый запуск через Docker Compose

```bash
docker compose up --build
```

После запуска сервис будет доступен по адресу `http://localhost:8080`.

Если `postgres` уже запускался ранее со старой схемой, пересоздай volume:

```bash
docker compose down -v
docker compose up --build
```

Причина в том, что SQL-файл из `migrations/0001_create_tasks.up.sql` монтируется в `docker-entrypoint-initdb.d` и применяется только при инициализации пустого data volume.

## API

Базовый префикс API:

```text
/api/v1
```

Основные маршруты:

- `POST /api/v1/tasks`
- `GET /api/v1/tasks`
- `GET /api/v1/tasks/{id}`
- `PUT /api/v1/tasks/{id}`
- `DELETE /api/v1/tasks/{id}`

## Предположения
1. Задачи повторяются бесконечно - если не указан repeat_until, задача будет создаваться вечно.
2. Часовой пояс utc - все даты и время хранятся в utc.
3. генератор проверяет задачи раз в минуту - компромисс между точностью и нагрузкой на бд.

## Ограничения 
1. Нельзя задать время окончания повторений с точностью до времени - repeat_until работает с точностью до дня
2. Нет поддержки часовых поясов - все задачи в utc.

## Реализованные типы периодичности 
Тип               Описание              Пример repeat_config
'daily'           каждый n-день         '{"interval": 2}'
'monthly'         определенное число    '{"day_of_month": 15}'
'parity'          четные\нечетные       '{"type": "even"}'
'specific_dates'  конкретные даты       '{"dates": ["2026-05-15", "2026-06-15"]}'

## Генерация повторяющихся задач

Фоновый процесс запускается каждую минуту, находит задачи с next_occurrence <=NOW(), создает копию задачи с parent_id ссылкой на оригинал, рассчитывает следующую дату выполнения, обновляет next_occurrence у родительской задачи, если задача достигла repeat_until - отключает периодичность.

## Архитектурные решения 

- почему jsonb для repeat_config?
  позволяет добавлять новые типы периодичности без изменения схемы БД
  гибкость настройки под разные сценарии
- Почему фоновый генератор?
  проще в реализации
  не требует внешних зависимостей
  легко тестировать
- Почему parent_id?
  позволяет отследить историю повторений
  каскадное удаление всей серии задач
